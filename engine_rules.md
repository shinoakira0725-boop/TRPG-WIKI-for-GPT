L2-13 轉移碼系統｜精簡完整格式
同步規則
世界共享：WORLD（事件）｜MAP（地圖）
個人獨立：PARTY（隊伍）｜BAG（背包）｜狀態（飢餓 / 疲勞）｜PLAYER_SKILL（玩家專屬技能）｜NPC（狀態與記憶）｜MISSION（任務紀錄與完成次數）
轉移碼原則
轉移碼不是一般UI；不得直接使用玩家UI當作轉移碼。
轉移碼採「精簡完整格式」：文字可壓縮，但核心欄位不得缺失。
輸出轉移碼時，只輸出 TRANSFER_CODE，不輸出劇情、不輸出一般UI、不輸出多餘說明。
資料結構
WORLD：WORLD_TICK / DAY_COUNT / 時段 / 天氣 / 當前事件 / 世界威脅
MAP：區域 / 類型 / 目前節點 / 已知連接節點
PLAYER_STATE：PLAYER_ID / 姓名 / 位置 / 飢餓數值+嚴重度 / 疲勞數值+嚴重度 / 金錢 / 徽章數 / 狀態 / PLAYER_SKILL
PARTY：固定6格；每格含寶可夢中英名 / 圖鑑編號 / 性別 / 屬性 / 主副性格 / 親密度 / 狀態 / 情緒 / 技能 / 戰鬥記憶 / 偏好行為與物品
BAG：金錢 / 徽章收納盒 / 物品名稱x數量；無物品寫空，不得只寫概略名稱
NPC_STATE：重要NPC的ID / 名稱 / 陣營 / 位置 / 關係 / 目標 / 狀態；無則寫無
NPC_MEMORY：互動記憶 / 戰鬥記憶 / 信任變化 / 最後相關事件；無則寫無
MISSION_STATE：任務ID / 類型 / 狀態 / 區域 / 目標 / 進度 / 關聯NPC / 備註
MISSION_STATS：搜尋 / 調查 / 捕捉 / 護送 / 黑市 / NPC戰術指導 完成次數
SNAPSHOT_LOCK：WORLD / MAP / PLAYER / PARTY / BAG / NPC / MISSION 均需 YES
完整性規則
不得只寫 Low / High / Stable，必須寫數值+嚴重度。
不得省略背包數量、徽章數、任務完成次數、NPC記憶、隊伍性格與親密度。
無資料時必須明確寫：無 / 0 / 空 / 未觸發。
若核心欄位缺失，讀取端必須拒絕載入，不得部分覆寫目前世界。

🎒【BAG完整性檢查鎖】
輸出 TRANSFER_CODE 前，必須對照最新 BAG 快照。
MONEY / BADGE_CASE / ITEMS 皆必填。
ITEMS 必須列出全部物品名x數量；無物品寫空。
禁止概略詞：道具若干 / 藥品類 / 已有物品 / 同上 / 省略。
曾出現且未消耗、丟棄、交易的物品不得遺漏。
若玩家提醒「背包有遺漏」，必須重查整份 BAG 並重輸出完整 TRANSFER_CODE，不得只補單項。
輸出前自檢：BAG_MONEY:YES / BADGE_CASE:YES / ITEM_NAME:YES / ITEM_COUNT:YES / NO_MISSING_ITEM:YES
BAG 缺漏時，TRANSFER_CODE 視為不完整，必須重新生成。

👥【多人模式補丁｜MULTIPLAYER_PATCH v1.0】
啟動條件：
1. 群組對話中偵測到大於一個角色卡登記名稱或 PLAYER_ID。
2. 玩家輸入「加入玩家 / 新玩家加入 / 多人模式」。
3. 轉移碼內 PLAYER_COUNT > 1。

一、角色卡登記名稱綁定 PLAYER_ID
1. 建立角色卡時，角色卡登記名稱必須綁定唯一 PLAYER_ID。
2. PLAYER_ID 為內部識別；玩家日常發言使用角色卡登記名稱即可。
3. 建議發言格式：【角色卡登記名稱】行動內容。
   例：【阿凱】我要往森林前進。
4. 若同名角色出現，必須要求更名或加後綴，不得讓兩位玩家共用同一登記名稱。
5. 登記名稱一旦綁定，PARTY / BAG / MONEY / BADGES / STATUS / SKILL 皆綁定該 PLAYER_ID。
6. 禁止用顯示暱稱、帳號名稱或上一位發言者推定玩家身份；多人狀態下未標示登記名稱且無法判斷時，必須要求補上名稱。

二、世界共通 / 玩家獨立
世界共通資料：
WORLD_TICK / DAY_COUNT / TIME / WEATHER / REGION / AREA / NODE / WORLD_EVENTS / MAP / NPC_STATE / NPC_MEMORY / MISSION_STATE。
玩家獨立資料：
PLAYER_STATE / PARTY / BAG / MONEY / BADGE_CASE / PLAYER_SKILL / STATUS / 個人任務進度 / 個人戰鬥記憶。
禁止將任一玩家的獨立資料覆蓋到其他玩家。

三、多人 Tick 結算
1. 同一回合收集所有已標示登記名稱的玩家行動後，WORLD_TICK 只增加 +1。
2. 結算順序：收集行動 → 驗證登記名稱 → 檢查衝突 → 依優先級結算 → 更新世界共通資料 → 更新各玩家獨立資料 → 輸出多人UI → Snapshot Lock。
3. 若玩家不在同一地點，場景描述必須分段，不得混在同一敘事中。
4. 若只有一位玩家輸入行動，可先結算該玩家；其他玩家視為未行動，不得自動代行。

四、行動衝突裁決
同一 Tick 內多名玩家作用於同一目標時：
1. 明確先輸入者優先。
2. 同時輸入時，合理性優先。
3. 合理性相同時，位置較近者優先。
4. 仍無法判斷時，輸出「衝突待裁決」，請玩家協調。
5. 未獲優先者不得被覆蓋，必須給出次級結果：干擾 / 延後 / 失敗 / 轉為輔助。
6. 交易、交換寶可夢、借用物品等跨玩家資源變動，必須雙方明確同意才成立。

五、多人對戰優先級
1. 已進入戰鬥的玩家行動優先於非戰鬥事件。
2. 同場戰鬥內依：技能優先度 → 速度/狀態 → 距離/位置 → 指令合理性 → 輸入先後 裁定。
3. 多名玩家攻擊同一目標時，必須分別結算，不得用後者覆蓋前者結果。
4. 多名玩家捕捉同一寶可夢時，同一 Tick 只允許一次主要捕捉判定；其餘玩家可輔助、牽制或干擾，但不得改寫主要捕捉結果。
5. 防禦、保護、閃避、救援等反應行動，可在合理情境下插入判定，但不得無條件取消已鎖定結果。
6. 戰鬥結果鎖定後不可回溯；多人行動不得作為改寫戰鬥結果的理由。

六、多人存檔鎖
輸出 TRANSFER_CODE 時，必須保存：
1. WORLD / MAP / NPC / MISSION 共通快照。
2. 所有 PLAYER_ID 的 PLAYER_STATE / PARTY / BAG / MONEY / BADGES / SKILL / STATUS。
3. 每位玩家 BAG 必須執行 BAG完整性檢查鎖。
4. 若任一玩家資料缺失，TRANSFER_CODE 視為不完整，必須重新生成或拒絕輸出。
5. 禁止只保存目前發話玩家而遺漏其他玩家。
